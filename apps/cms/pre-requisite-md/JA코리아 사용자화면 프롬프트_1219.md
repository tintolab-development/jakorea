U-02-02. 신청 완료(승인 대기)

Follow the global rules defined in \_rules.prompt without exception.

REFERENCE:

* Domain Context: user\_domain\_context.prompt (v1.0)
* Common State / Enum Definition Document
* Screen–API Mapping Table
* Screen Wireframe: U-02-02 신청 완료(승인 대기) 화면

ROLE:
당신은 JA Korea 교육업무 자동화 서비스의
사용자 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.
이 화면은 사용자가 신청 직후 느낄 수 있는
불안·혼란·추측을 완전히 제거하는 것을
최우선 목표로 하는 화면입니다.

SCREEN ID:
U-02-02

SCREEN NAME:
신청 완료 (승인 대기)

SCREEN PURPOSE:

* 사용자가 신청이 정상적으로 접수되었음을 확신한다.
* 현재 상태가 “관리자 승인 대기”임을 명확히 이해한다.
* 지금은 추가 행동이 필요 없다는 점을 분명히 인지한다.
* 불필요한 재신청, 수정, 문의를 구조적으로 차단한다.

INPUT DATA (API RESPONSE):

* applicationId: string
* applicationStatus: ApplicationStateCode (APPLY\_01)
* program:

  * programId: string
  * programName: string

* submittedAt: datetime
* reason\_public?: string (항상 null)

ALLOWED API USAGE:

* GET /applications/{applicationId}

FORBIDDEN:

* 상태(Enum) 추론 또는 생성
* 승인/확정으로 오해될 수 있는 표현
* 수정/취소/재신청 버튼 노출
* 관리자 관련 정보 노출
* 복수 주요 CTA 동시 노출

UI STRUCTURE REQUIREMENTS:

1. 상태 확인 영역 (최상단, 가장 강조)

* 상태 아이콘 + 상태 제목을 함께 표시
* 상태 제목 문구:
  “신청이 정상적으로 접수되었습니다”
* 이 영역에서 “확정”이라는 단어 사용 금지

2. 상태 설명 영역

* 반드시 아래 두 문장을 그대로 포함할 것:

  1. “현재 관리자 승인 대기 상태입니다.”
  2. “현재 추가로 하실 일은 없습니다.”

* 문장 순서 변경 불가
* 의미를 변형하거나 요약하지 말 것

3. 신청 요약 영역

* 프로그램명 표시
* 신청 일시 표시
* 신청 ID는 사용자에게 노출하지 않음

4. 다음 행동 안내 영역

* 사용자가 이동할 수 있는 CTA는 단 하나만 허용
* CTA:
  \[마이페이지로 이동]
* 보조 설명 문구:
  “승인 결과는 마이페이지에서 확인하실 수 있습니다.”

5. 보조 안내 영역 (조건부)

* 승인 절차에 대한 추가 설명이 필요한 경우
  다음 문구를 그대로 노출:
  “승인 결과는 문자 및 서비스 알림으로도 안내드립니다.”

STATE \& UX RULES:

* applicationStatus는 반드시 APPLY\_01 상태로 고정
* reason\_public은 이 화면에서 사용되지 않음
* 이 화면에서 사용자가 수행할 수 있는
  유효한 액션은 ‘이동’뿐이다
* 사용자가 ‘기다림’을 불안으로 인식하지 않도록
  모든 문구는 안정적인 어조를 유지해야 한다

ERROR HANDLING:

* API 오류 시 기술적 정보 노출 금지
* 표준 오류 문구 사용:
  “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”

DELIVERABLE:

* React Functional Component
* TypeScript
* Props 인터페이스 명확히 정의
* 상태/문구는 하드코딩이 아닌 Enum 기반 조건으로 제어
* 비즈니스 로직은 컴포넌트 외부로 분리

U-02-03. 신청 반려

Follow the global rules defined in \_rules.prompt without exception.

REFERENCE:

* Domain Context: user\_domain\_context.prompt (v1.0)
* Common State / Enum Definition Document
* Screen–API Mapping Table
* Screen Wireframe: U-02-03 신청 반려 화면

ROLE:
당신은 JA Korea 교육업무 자동화 서비스의
사용자 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.
이 화면은 “신청이 승인되지 않은 상황”을
사용자가 감정적 오해 없이 이해하도록 돕는
매우 민감한 결과 화면입니다.

SCREEN ID:
U-02-03

SCREEN NAME:
신청 반려

SCREEN PURPOSE:

* 사용자가 신청이 반려되었음을 명확히 인지한다.
* 반려 사유(reason\_public)를 오해 없이 정확히 이해한다.
* 반려가 오류나 실패가 아닌 ‘결과’임을 받아들이게 한다.
* 불필요한 문의, 항의, 재시도를 UX 차원에서 차단한다.
* 가능한 다음 경로가 있다면 명확히 안내한다.

INPUT DATA (API RESPONSE):

* applicationId: string
* applicationStatus: ApplicationStateCode (APPLY\_03)
* program:

  * programId: string
  * programName: string

* submittedAt: datetime
* reason\_public: string (필수, 1문장, 최대 50자)

ALLOWED API USAGE:

* GET /applications/{applicationId}

FORBIDDEN:

* 상태(Enum) 추론 또는 생성
* 반려를 완화하거나 평가하는 표현
* 관리자 판단에 대한 설명·해명·정당화
* 수정/재신청 버튼 자동 노출
* 문의 유도 문구 (“문의하세요”, “담당자 연락” 등)
* 복수 주요 CTA 동시 노출

UI STRUCTURE REQUIREMENTS:

1. 상태 결과 영역 (최상단, 가장 강조)

* 상태 아이콘 + 상태 제목을 함께 표시
* 상태 제목 문구:
  “신청이 반려되었습니다”
* ‘실패’, ‘오류’, ‘문제’ 등의 단어 사용 금지

2. 반려 사유 영역 (핵심)

* reason\_public을 반드시 단독 영역으로 표시
* 제목 예시:
  “반려 사유”
* reason\_public은:

  * 그대로 출력 (의미 변경·요약·재작성 금지)
  * 1문장 그대로 노출

* reason\_public이 null 또는 빈 값일 경우
  반드시 아래 문구를 대신 노출:
  “사유를 확인할 수 없습니다.”

3. 신청 요약 영역

* 프로그램명 표시
* 신청 일시 표시
* 신청 ID는 사용자에게 노출하지 않음

4. 다음 행동 안내 영역

* 사용자가 선택할 수 있는 CTA는 최대 1개
* 기본 CTA:
  \[다른 프로그램 보기]
* 조건부 CTA(정책상 허용된 경우에만):
  \[다시 신청하기]
  ※ 이 CTA는 반드시
  “재신청이 가능한 경우”에만 노출
  (프론트엔드에서 임의 판단 금지)

5. 보조 안내 영역

* 아래 문구 중 하나를 상황에 맞게 노출:

  * “반려된 신청은 자동으로 종료 처리됩니다.”
  * “다른 프로그램에는 정상적으로 신청하실 수 있습니다.”

STATE \& UX RULES:

* applicationStatus는 반드시 APPLY\_03 상태여야 한다.
* reason\_public은 이 화면에서 필수이며,
  표시되지 않으면 화면은 실패로 간주한다.
* 사용자가 이 결과를 “개인적 평가”로
  받아들이지 않도록 중립적 어조를 유지한다.
* 이 화면에서 사용자는
  ‘이해 → 이동’ 외의 행동을 요구받지 않는다.

ERROR HANDLING:

* API 오류 시 기술적 정보 노출 금지
* 표준 오류 문구 사용:
  “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”

DELIVERABLE:

* React Functional Component
* TypeScript
* Props 인터페이스 명확히 정의
* reason\_public 출력 로직은 조건문으로 명시적으로 처리
* 비즈니스 판단(재신청 가능 여부)은
  컴포넌트 외부에서 결정된 값만 사용

\- Screen Wireframe: U-02-04 신청 승인 완료 화면



ROLE:

당신은 JA Korea 교육업무 자동화 서비스의

사용자 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.

이 화면은 “신청 → 승인”이라는 중요한 전이를

사용자가 명확하고 안정적으로 인식하도록 돕는

결과 중심 화면입니다.



SCREEN ID:

U-02-04



SCREEN NAME:

신청 승인 완료



SCREEN PURPOSE:

\- 사용자가 자신의 신청이 승인되었음을 명확히 인지한다.

\- 승인 결과가 ‘확정’임을 오해 없이 이해한다.

\- 이후 무엇이 진행될지(일정/활동)를 예측 가능하게 안내한다.

\- 다음 화면으로의 이동을 단 하나의 명확한 경로로 유도한다.



INPUT DATA (API RESPONSE):

\- applicationId: string

\- applicationStatus: ApplicationStateCode (APPLY\_02)

\- program:

&nbsp; - programId: string

&nbsp; - programName: string

\- approvedAt: datetime

\- nextStep:

&nbsp; - type: SCHEDULE | ACTIVITY | NONE

&nbsp; - targetUrl?: string



ALLOWED API USAGE:

\- GET /applications/{applicationId}



FORBIDDEN:

\- 상태(Enum) 추론 또는 생성

\- 승인 상태를 축소·완화하는 표현

\- 관리자 판단 과정에 대한 설명

\- 승인 취소/수정/재신청 버튼 노출

\- 복수 주요 CTA 동시 노출



UI STRUCTURE REQUIREMENTS:



1\. 승인 결과 영역 (최상단, 가장 강조)

\- 상태 아이콘 + 상태 제목을 함께 표시

\- 상태 제목 문구:

&nbsp; “신청이 승인되었습니다”

\- 이 영역에서는 반드시

&nbsp; ‘승인’, ‘확정’이라는 의미가 분명히 전달되어야 한다.



2\. 승인 설명 영역

\- 아래 문장을 반드시 포함할 것:

&nbsp; “신청하신 프로그램 참여가 확정되었습니다.”

\- 승인 완료가 ‘과정 중 하나’가 아니라

&nbsp; ‘결과’임을 명확히 인식시키는 어조 유지



3\. 프로그램 요약 영역

\- 프로그램명 표시

\- 승인 일시 표시

\- 불필요한 세부 정보 노출 금지



4\. 다음 단계 안내 영역 (핵심)

\- nextStep.type 값에 따라 분기 렌더링



&nbsp; 4-1. nextStep.type === SCHEDULE

&nbsp; - 안내 문구 예시:

&nbsp;   “확정된 일정은 마이페이지에서 확인하실 수 있습니다.”

&nbsp; - CTA:

&nbsp;   \[마이페이지로 이동]



&nbsp; 4-2. nextStep.type === ACTIVITY

&nbsp; - 안내 문구 예시:

&nbsp;   “이제 프로그램 활동이 시작됩니다.”

&nbsp; - CTA:

&nbsp;   \[활동 상세 보기]



&nbsp; 4-3. nextStep.type === NONE

&nbsp; - 안내 문구 예시:

&nbsp;   “추가 안내가 있을 경우 알려드리겠습니다.”

&nbsp; - CTA:

&nbsp;   \[마이페이지로 이동]



\- CTA는 반드시 1개만 노출

사용자 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.

이 화면은 사용자가 프로그램을 정확히 이해하고,

신청 여부를 판단하는 가장 중요한 기준 화면입니다.



SCREEN ID:

U-01-02



SCREEN NAME:

프로그램 상세



SCREEN PURPOSE:

\- 사용자가 프로그램의 목적과 내용을 오해 없이 이해한다.

\- 모든 프로그램이 승인제 기반임을 자연스럽게 인지한다.

\- 신청 가능 여부를 추측하지 않고 명확히 인식한다.

\- 신청 전 불필요한 문의와 혼란을 구조적으로 차단한다.



INPUT DATA (API RESPONSE):

\- programId: string

\- programName: string

\- category: string

\- description: string

\- targetAudience: Array of UserRole (STUDENT | TEACHER)

\- programStatus: ProgramState (PROG\_01 | PROG\_02 | PROG\_03)

\- applicationAvailability:

&nbsp; - canApply: boolean

&nbsp; - reason\_public?: string

\- scheduleSummary?: Array of

&nbsp; - dateRange: string

&nbsp; - status: ScheduleStatus (SCH\_01 | SCH\_02 | SCH\_03)

\- requiresInstructor: boolean

\- requiresVolunteer: boolean



ALLOWED API USAGE:

\- GET /programs/{programId}



FORBIDDEN:

\- 프로그램 상태를 프론트엔드에서 추론하거나 생성하는 행위

\- 신청 가능 여부를 클라이언트 로직으로 판단하는 행위

\- 승인제 설명을 누락하거나 축소하는 표현

\- 미확정 일정 또는 내부 운영 일정 노출

\- 신청 버튼을 조건 없이 노출

\- 관리자/운영 정보 노출



UI STRUCTURE REQUIREMENTS:



1\. 프로그램 핵심 정보 영역 (상단)

\- 프로그램명 표시

\- 카테고리 표시

\- 프로그램 목적 및 설명을 문장 중심으로 표시

\- 홍보 문구가 아닌 ‘이해용 설명’ 톤 유지



2\. 대상 및 참여 방식 안내 영역

\- 대상 사용자 명확히 문장으로 표시

&nbsp; 예: “이 프로그램은 학생을 대상으로 합니다.”

\- 강사/봉사자 참여 필요 여부를

&nbsp; 설명 문장으로 안내

\- 내부 운영 기준처럼 보이는 용어 사용 금지



3\. 일정 요약 영역 (조건부)

\- scheduleSummary가 존재할 경우에만 노출

\- 확정된 일정만 요약 형태로 표시

\- 일정이 없는 경우 아래 문구를 반드시 노출:

&nbsp; “일정은 승인 후 확정되어 안내됩니다.”



4\. 신청 안내 및 상태 영역 (핵심)

\- 승인제 고정 문구를 반드시 포함:

&nbsp; “본 프로그램은 신청 후 관리자 승인을 통해 참여가 확정됩니다.”



\- applicationAvailability.canApply === true

&nbsp; - CTA:

&nbsp;   \[프로그램 신청하기]



\- applicationAvailability.canApply === false

&nbsp; - CTA 미노출

&nbsp; - reason\_public이 있을 경우 그대로 노출

&nbsp; - reason\_public이 없을 경우 아래 문구 노출:

&nbsp;   “현재 신청이 불가능한 상태입니다.”



5\. 보조 안내 영역 (v1.1 패치 반영)

\- 아래 문구를 정보성으로 1회 노출:

&nbsp; “모든 프로그램은 신청 후 승인 절차를 거쳐 참여가 확정됩니다.”



STATE \& UX RULES:

\- 이 화면에서는 신청 상태(APPLY\_xx)를 노출하지 않는다.

\- 신청 가능 여부는 반드시 서버 응답을 그대로 사용한다.

\- CTA는 최대 1개만 노출한다.

\- 사용자가 ‘신청해도 되는지’를

&nbsp; 추측하게 만드는 UI는 실패다.



ERROR HANDLING:

\- API 오류 시 기술적 정보 노출 금지

\- 표준 오류 문구 사용:

&nbsp; “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”



DELIVERABLE:

\- React Functional Component

\- TypeScript

\- Props 인터페이스 명확히 정의

\- 신청 가능 여부 렌더링은 명시적 조건문으로 구현

\- 비즈니스 판단 로직은 컴포넌트 외부에서 처리

INPUT DATA (API RESPONSE):

\- applicationId: string

\- applicationStatus: ApplicationStateCode (APPLY\_03)

\- program:

&nbsp; - programId: string

&nbsp; - programName: string

\- submittedAt: datetime

\- reason\_public: string (필수, 1문장, 최대 50자)



ALLOWED API USAGE:

\- GET /applications/{applicationId}



FORBIDDEN:

\- 상태(Enum) 추론 또는 생성

\- 반려를 완화하거나 평가하는 표현

\- 문의 유도 문구

&nbsp; (예: “문의하세요”, “담당자에게 연락”)

\- 복수 주요 CTA 동시 노출



UI STRUCTURE REQUIREMENTS:



1\. 상태 결과 영역 (최상단, 가장 강조)

\- 상태 아이콘 + 상태 제목을 함께 표시

\- 상태 제목 문구:

&nbsp; “신청이 반려되었습니다”

\- ‘실패’, ‘오류’, ‘문제’ 등의 단어 사용 금지

\- 결과임을 담담하게 전달하는 어조 유지



2\. 반려 사유 영역 (핵심)

\- 제목:

&nbsp; “반려 사유”

\- reason\_public을 반드시 단독 영역으로 표시

\- reason\_public은:

&nbsp; - 서버 응답 그대로 출력

&nbsp; - 의미 변경, 요약, 재작성 금지

\- reason\_public이 null 또는 빈 값일 경우

&nbsp; 반드시 아래 문구를 대신 노출:

&nbsp; “사유를 확인할 수 없습니다.”



3\. 신청 요약 영역

\- 프로그램명 표시

\- 신청 일시 표시

\- 신청 ID는 사용자에게 노출하지 않음



4\. 다음 행동 안내 영역

\- 사용자가 선택할 수 있는 CTA는 최대 1개



\- 기본 CTA:

&nbsp; \[다른 프로그램 보기]



\- 조건부 CTA (정책상 허용된 경우에만):

&nbsp; \[다시 신청하기]

&nbsp; ※ 이 CTA는 반드시

&nbsp;    “재신청이 허용된 경우”에만 노출

&nbsp;    (프론트엔드 임의 판단 금지)



5\. 보조 안내 영역

\- 아래 문구 중 하나를 상황에 맞게 노출:

&nbsp; - “반려된 신청은 자동으로 종료 처리됩니다.”

&nbsp; - “다른 프로그램에는 정상적으로 신청하실 수 있습니다.”



STATE \& UX RULES:

\- applicationStatus는 반드시 APPLY\_03 상태여야 한다.

\- reason\_public은 필수이며,

&nbsp; 이 값이 화면에 표시되지 않으면 실패로 간주한다.

\- 사용자가 이 결과를

&nbsp; ‘개인적 평가’나 ‘부정적 피드백’으로

&nbsp; 오해하지 않도록 중립적 어조를 유지한다.

\- 이 화면에서 사용자는

&nbsp; ‘이해 → 이동’ 외의 행동을 요구받지 않는다.



ERROR HANDLING:

\- API 오류 시 기술적 정보 노출 금지

\- 표준 오류 문구 사용:

&nbsp; “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”



DELIVERABLE:

\- React Functional Component

\- TypeScript

\- Props 인터페이스 명확히 정의

&nbsp; 조건문으로 명시적으로 처리

\- 재신청 가능 여부 판단은

&nbsp; 컴포넌트 외부에서 전달된 값만 사용

\- 모든 프로그램이 승인제 기반임을 선제적으로 인지한다.

U-04-01 강의 상세 화면



Follow the global rules defined in \_rules.prompt without exception.



REFERENCE:

\- Domain Context: user\_domain\_context.prompt (v1.1)

\- Common State / Enum Definition Document

\- Screen–API Mapping Table

\- Screen Wireframe: U-04-01 강의 상세 화면



ROLE:

당신은 JA Korea 교육업무 자동화 서비스의

사용자(강사) 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.

이 화면은 강사가 본인의 강의 활동을 수행하기 위한

행동 중심(Activity) 상세 화면이며,

현재 상태와 다음 행동을 추측 없이 이해시키는 것이 목적입니다.



SCREEN ID:

U-04-01



SCREEN NAME:

강의 상세



SCREEN PURPOSE:

\- 강사가 현재 강의 상태를 즉시 이해한다.

\- 강의 일정/장소/진행 방식 등 필수 정보를 정확히 확인한다.

\- 지금 해야 할 행동(없음/완료 처리/보고서)을 혼동 없이 안내한다.

\- 강의 완료 후 보고서 및 정산 흐름으로 자연스럽게 연결한다.



INPUT DATA (API RESPONSE):

\- lectureId: string

\- lectureStatus: LectureStatus (LECT\_01 | LECT\_02 | LECT\_03)

\- program:

&nbsp; - programId: string

&nbsp; - programName: string

\- schedule:

&nbsp; - date: string

&nbsp; - timeRange: string

&nbsp; - mode: ONLINE | OFFLINE

&nbsp; - location?: string

&nbsp; - joinUrl?: string

\- roleDescription: string

\- nextRequiredAction:

&nbsp; - type: NONE | COMPLETE | REPORT

&nbsp; - targetUrl?: string

\- reason\_public?: string



ALLOWED API USAGE:

\- GET /lectures/{lectureId}

\- POST /lectures/{lectureId}/complete



FORBIDDEN:

\- 강의 상태(Enum) 추론 또는 생성

\- 강의 완료 자동 처리

\- 보고서 화면으로 자동 이동

\- 관리자/운영 정보 노출 (reason\_internal 등)

\- 복수 주요 CTA 동시 노출

\- “문의하세요/연락하세요” 문구



UI STRUCTURE REQUIREMENTS:



1\. 강의 상태 요약 영역 (최상단, 가장 강조)

\- lectureStatus 기반으로 단 하나의 상태 문장 표시

&nbsp; - LECT\_01:

&nbsp;   “강의가 예정되어 있습니다.”

&nbsp;   (v1.1 필수 추가 문구)

&nbsp;   “현재 활동은 아직 시작되지 않았습니다.

&nbsp;    활동이 시작되면 이 화면에서 안내드립니다.”

&nbsp; - LECT\_02:

&nbsp;   “현재 강의가 진행 중입니다.”

&nbsp; - LECT\_03:

&nbsp;   “강의가 완료되었습니다.”

\- 상태는 반드시 텍스트로 이해 가능해야 한다.

\- reason\_public이 필요한 상황(변경/취소 등)에서 값이 있으면 그대로 표시

&nbsp; (의미 변경/요약 금지)



2\. 프로그램 및 일정 정보 영역

\- 프로그램명 표시

\- 강의 일자/시간 표시

\- 진행 방식 표시 (온라인/오프라인)

\- 오프라인인 경우: 장소(location) 표시

\- 온라인인 경우: 참여 링크(joinUrl)는

&nbsp; 읽기 정보로만 표시(CTA로 노출하지 않음)



3\. 역할 및 수행 안내 영역

\- roleDescription을 문장으로 그대로 표시

\- 사용자가 “내가 무엇을 해야 하는 역할인지” 이해 가능해야 한다



4\. 다음 행동 안내 영역 (핵심)

\- nextRequiredAction.type 값에 따라 분기

&nbsp; - NONE:

&nbsp;   “현재 추가로 하실 일은 없습니다.”

&nbsp;   (CTA 없음)

&nbsp; - COMPLETE:

&nbsp;   안내 문구:

&nbsp;   “강의 종료 후 완료 처리를 진행해 주세요.”

&nbsp;   CTA(단일):

&nbsp;   \[강의 완료 처리]

&nbsp;   → POST /lectures/{lectureId}/complete 호출

&nbsp; - REPORT:

&nbsp;   안내 문구:

&nbsp;   “강의보고서 제출이 필요합니다.”

&nbsp;   CTA(단일):

&nbsp;   \[보고서 작성하기]

&nbsp;   → targetUrl로 이동



\- 어떤 경우에도 CTA는 1개만 노출



5\. 보조 안내 영역 (v1.1 정합성 유지)

\- 아래 문구를 정보성으로 1회 노출:

&nbsp; “강의 완료 후 보고서 제출 및 정산 절차가 진행됩니다.”



STATE \& UX RULES:

\- lectureStatus 및 nextRequiredAction은 서버 응답을 그대로 사용한다.

\- 완료/보고서 제출은 강사의 명시적 행동이 있어야만 가능하다.

\- 사용자가 “지금 뭘 해야 하지?”를 고민하면 실패다.

\- 이 화면은 ‘관리’가 아니라 ‘수행’을 돕는 화면이다.



ERROR HANDLING:

\- API 오류 시 기술적 정보 노출 금지

\- 표준 오류 문구 사용:

&nbsp; “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”



DELIVERABLE:

\- React Functional Component

\- TypeScript

\- Props 인터페이스 명확히 정의

\- 상태/CTA 분기는 명시적 조건문으로 구현

\- 완료 처리 API 호출은 컴포넌트 외부 핸들러로 분리

\- 네비게이션은 targetUrl 기반으로만 수행

이력 목록

Follow the global rules defined in \_rules.prompt without exception.



REFERENCE:

\- Screen Wireframe: U-05-02 이력 상세 / 증빙 화면



ROLE:

당신은 JA Korea 교육업무 자동화 서비스의

사용자 화면을 개발하는 시니어 프론트엔드 엔지니어입니다.

이 화면은 사용자의 참여/활동 결과가

공식 기록으로 확정된 이후 이를 확인하고

증빙 문서를 내려받는 최종 결과(Result) 화면입니다.



SCREEN ID:

U-05-02



SCREEN NAME:

이력 상세 / 증빙



SCREEN PURPOSE:

\- 사용자가 자신의 참여 결과를 정확히 이해한다.

\- 이력이 공식 기록이며 수정 불가함을 인지한다.

\- 필요한 증빙 문서를 신뢰하고 다운로드한다.

\- 추가 행동이 필요 없음을 명확히 인식한다.



INPUT DATA (API RESPONSE):

\- historyId: string

\- historyType: PROGRAM | LECTURE | VOLUNTEER

\- finalStatus:

&nbsp; - ProgramState (PROG\_03)

&nbsp; - LectureStatus (LECT\_03)

&nbsp; - VolunteerActivityStatus (VOL\_03)

\- program:

&nbsp; - programId: string

&nbsp; - programName: string

\- participation:

&nbsp; - role: UserRole (STUDENT | TEACHER | INSTRUCTOR | VOLUNTEER)

&nbsp; - completedAt: datetime

\- resultSummary:

&nbsp; - instructorSettlement?:

&nbsp;   - settlementStatus: SettlementStatus (PAY\_03 | PAY\_04)

&nbsp;   - paidAmount?: number

&nbsp; - volunteerHours?:

&nbsp;   - hours: number

\- certificates: Array of

&nbsp; - type: CERTIFICATE | STATEMENT

&nbsp; - title: string

&nbsp; - downloadUrl: string



ALLOWED API USAGE:

\- GET /me/histories/{historyId}

\- GET /volunteers/certificates/{historyId}

\- GET /settlements/{historyId}/statement



FORBIDDEN:

\- 상태(Enum) 변경 또는 암시

\- 수정/삭제/재요청/재계산 버튼 노출

\- 관리자 판단 과정 설명

\- “문의하세요/담당자 연락” 문구

\- 복수 주요 CTA 동시 노출



UI STRUCTURE REQUIREMENTS:



1\. 결과 요약 영역 (최상단, 가장 강조)

\- finalStatus를 기준으로 단 하나의 결과 문장 표시

&nbsp; 예:

&nbsp;   - PROG\_03:

&nbsp;     “프로그램 참여가 완료되었습니다.”

&nbsp;   - LECT\_03:

&nbsp;     “강의가 정상적으로 완료되었습니다.”

&nbsp;   - VOL\_03:

&nbsp;     “봉사가 완료되었으며 봉사시간이 확정되었습니다.”

\- ‘완료’, ‘확정’, ‘공식 기록’의 의미가 명확히 전달되어야 한다.



2\. 프로그램 및 참여 정보 영역

\- 프로그램명 표시

\- 참여 역할 표시

\- 완료 일시 표시

\- 읽기 전용 UI



3\. 결과 상세 요약 영역 (조건부)

\- participation.role === INSTRUCTOR 인 경우

&nbsp; - 정산 상태 문구 표시

&nbsp;   예:

&nbsp;     - PAY\_03: “정산이 승인되었습니다.”

&nbsp;     - PAY\_04: “지급이 완료되었습니다.”

&nbsp; - 지급 완료(PAY\_04)인 경우

&nbsp;   지급 금액 표시

\- participation.role === VOLUNTEER 인 경우

&nbsp; - 봉사시간 표시:

&nbsp;   “인정 봉사시간: 총 {hours}시간”

&nbsp; - 아래 문구를 반드시 함께 노출:

&nbsp;   “봉사시간은 프로그램 기준 고정 시간으로 인정됩니다.”



4\. 증빙 문서 영역 (핵심)

\- certificates 배열 기준으로 렌더링

\- 각 증빙 항목에 대해:

&nbsp; - title 표시

&nbsp; - 단일 CTA 제공:

&nbsp;   \[다운로드]

\- 증빙 예:

&nbsp; - 수료 확인서

&nbsp; - 봉사 확인서

&nbsp; - 지급조서(PDF)



5\. 공식 기록 안내 영역 (v1.1 패치 반영)

\- 아래 문구를 반드시 고정 노출:

&nbsp; “본 이력 및 증빙 문서는

&nbsp;  공식 기록으로 보관되며 수정할 수 없습니다.”



6\. 완료 상태 안내 영역

\- 반드시 아래 문구 중 하나를 노출:

&nbsp; - “추가로 하실 일은 없습니다.”

&nbsp; - “이 결과는 최종 확정 상태입니다.”



STATE \& UX RULES:

\- 이 화면은 읽기 전용이다.

\- finalStatus는 서버 응답을 그대로 사용한다.

\- 사용자가 이 화면에서

&nbsp; “무언가 더 해야 하나?”라고 고민하면 실패다.

\- 모든 결과는 ‘되돌릴 수 없음’을 UX로 명확히 전달해야 한다.



ERROR HANDLING:

\- API 오류 시 기술적 정보 노출 금지

\- 표준 오류 문구 사용:

&nbsp; “처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요.”

\- TypeScript

\- 결과/증빙 분기는 명시적 조건문으로 구현

\- 다운로드는 단순 링크 호출로 처리
